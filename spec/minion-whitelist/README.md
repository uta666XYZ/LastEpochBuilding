## Runtime capture procedure

1. Install plugin: `LE_datamining/bepinex/MinionStatDumper/bin/Release/MinionStatDumper.dll`
   → `<LE>/BepInEx/plugins/`. (Already copied.)
2. Launch LE with a Necromancer / Beastmaster / Falconer character at lv ≥ 80.
3. Summon a full roster of minions (skeletons, golem, wraiths, companion, etc.).
4. Equip / unequip a few minion affixes to broaden the observation set.
5. Quit the game cleanly — final dump fires on `Application.quitting`.
   (Periodic 30s dumps + 2s scans land observations meanwhile.)
6. Output lands at `%USERPROFILE%/Documents/LE_datamining/extracted/minion_runtime_stats.json`.
7. Copy it next to the two static surfaces in this directory.

## LEB wire-in — DONE (2026-05-17)

The provenance for "which (sp, tags, specialTag) tuples route to minions"
lives here. The actual runtime routing on the LEB side is in
`src/Modules/CalcDefence.lua` (~line 1735, the `sumMinion` bucket loop —
guarded by `@leb-regression-guard: minion-whitelist-3surface-union`).
The bucket walks `MinionModifier` LIST entries emitted by
`src/Modules/ModParser.lua` and `src/Data/SkillStatMap.lua`; this
whitelist is the audit trail for what those upstream emitters should be
recognising.

Coverage spec: `MinionWhitelistCoverage_spec.lua` (this dir) — asserts
the file loads, every entry has a non-empty `surfaces` list, the
confidence tier matches surface count, and the CalcDefence sumMinion
block keeps the guard marker referencing this file.

## Tags reference

| Bit | Meaning |
|---|---|
| 1 | Physical |
| 2 | Lightning |
| 4 | Cold |
| 8 | Fire |
| 16 | Void |
| 32 | Necrotic |
| 64 | Poison |
| 8192 | **MINION** (route the stat to minions) |

Composite examples: `8193` = Physical Minion Damage, `8200` = Fire Minion
Damage, `8194` = Lightning Minion Damage.
